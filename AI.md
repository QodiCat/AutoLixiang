

# AI 协作需求平台（React 前端模拟版）需求文档

## 1. 项目目标

打造一个多角色协作平台，验证从“业务提目标”到“方案产出与汇报”的 AI 协作闭环。  
项目仅使用 React 前端实现，数据和接口全部使用模拟数据，不接入真实后端。

## 2. 版本范围

### 2.1 MVP 目标

- 支持 4 个角色登录并进入差异化工作台。
- 支持业务需求输入、Agent 追问澄清、产物生成、管理员汇总展示的完整流程。
- 支持核心产物可视化展示：优化需求、PRD 草案、技术方案草案、HTML 汇报页。

### 2.2 本期不做

- 不做真实账号体系（注册、组织、SSO）。
- 不做真实第三方接入（飞书、行业报告 API、竞品抓取）。
- 不做后端存储和持久化。

## 3. 角色与职责

### 3.1 业务方

- 输入碎片化目标、用户原声、会议纪要、约束条件。
- 回答 Agent 多轮追问。
- 确认优化后的需求版本。

### 3.2 产品同学

- 查看业务原始需求与 Agent 优化结果。
- 触发生成 PRD 草案与原型草案。
- 审核功能模块、业务流程、验收标准、计划与风险。

### 3.3 技术同学

- 基于业务与产品结果生成技术方案草案。
- 审核可行性、系统边界、风险与排期建议。

### 3.4 管理员

- 汇总全角色关键信息。
- 生成并查看面向业务方的可视化 HTML 汇报页。

## 4. 核心流程

1. 业务方创建需求并输入上下文材料。  
2. Agent 发起多轮追问，补全背景、约束、成功标准。  
3. 系统生成“优化需求结果”，业务方确认。  
4. 产品同学基于需求结果生成 PRD/原型草案并审核。  
5. 技术同学基于最终产品内容生成技术方案草案并审核。  
6. 管理员整合信息，输出可视化汇报页。

## 5. 功能需求

### 5.1 登录与角色切换（模拟）

- 提供固定模拟账号入口：业务方、产品同学、技术同学、管理员。
- 登录后进入对应角色首页，导航和模块按角色隔离。

### 5.2 业务方工作台

- 新建需求任务。
- 输入方式：文本输入、粘贴会议纪要、粘贴文档内容（模拟上传）。
- Agent 追问区：多轮问答（可继续追问、可结束会话）。
- 输出结果区：展示优化后的需求结构化结果。

### 5.3 产品同学工作台

- 查看“原始需求”和“优化需求”。
- 一键生成 PRD 草案（模拟）。
- 一键生成原型草案（模拟页面/模块图）。
- 展示执行计划、资源预算、风险评估。

### 5.4 技术同学工作台

- 读取产品最终提交内容。
- 一键生成技术方案草案。
- 展示章节：技术架构、模块边界、实现建议、风险点、排期建议。

### 5.5 管理员工作台

- 展示需求全景：当前阶段、关键结论、风险状态。
- 一键生成 HTML 汇报页（模拟）。
- 可按任务查看业务价值、方案摘要、计划与风险。

## 6. 关键数据对象（前端模拟）

### 6.1 需求任务 Demand

- `id`
- `title`
- `creatorRole`
- `status`
- `createdAt`
- `updatedAt`

### 6.2 输入材料 InputItem

- `id`
- `demandId`
- `type`（text/meeting_note/doc）
- `content`

### 6.3 追问会话 AgentSession

- `id`
- `demandId`
- `rounds`（问答列表）
- `summary`

### 6.4 产物 Artifact

- `id`
- `demandId`
- `type`（optimized_requirement/prd/prototype/tech_solution/report_html）
- `version`
- `content`
- `ownerRole`

## 7. 状态流转

- `draft`：草稿创建  
- `clarifying`：Agent 追问中  
- `optimized`：需求已优化  
- `prd_ready`：PRD/原型已生成  
- `tech_ready`：技术方案已生成  
- `report_ready`：汇报页已生成  
- `archived`：归档

## 8. 页面清单

- 登录页（角色选择）
- 业务方首页（任务列表）
- 业务方任务详情（输入、追问、优化结果）
- 产品同学首页（任务列表）
- 产品任务详情（需求对比、PRD/原型、计划风险）
- 技术同学首页（任务列表）
- 技术任务详情（技术方案）
- 管理员首页（全局看板）
- 管理员任务汇报页（HTML 预览）

## 9. 交互与体验要求

- 每个角色只看到与职责相关的功能入口。
- 同一任务在不同角色视角下展示不同信息密度。
- 关键操作有明确反馈：生成中、成功、失败（模拟）。
- 主要页面在桌面端可用，移动端至少保证基础浏览可读。

## 10. 成功评估标准（MVP）

- 交互深度：每个任务至少完成 3 轮以上高质量追问。
- 价值溢出：优化需求中至少新增 1 条业务方未显式提出的价值点。
- 落地程度：PRD 与技术方案均包含可执行的验收项与风险项。
- 角色转变：产品同学工作台中“审核/决策”区域使用时长高于“纯编辑”区域。

## 11. 技术实现约束

- 技术栈：React（可搭配 TypeScript）。
- 数据来源：本地 mock 文件或前端内存状态。
- 不依赖后端服务，不发起真实外网检索。
- 所有“AI 生成”均由前端模拟数据和状态机驱动。
