以下是该网站的逻辑架构与核心要点梳理，只用react撰写，不需要后端【后端模拟数据即可】：

------

## 1. 核心定位：从“工具辅助”到“新协作范式”


- **传统模式**：业务提需求 -> 产品面谈 -> 手写 PRD -> 技术评审。
- **新协作范式**：业务提目标 -> **Agent 实时追问/知识整合/方案推演** -> 自动生成可交互原型与执行计划 -> 产品经理进行价值审核与编排。


------

## 2. 功能要求：

这个网站要求可以以多个角色登录【模拟数据】每个角色登录进去后界面不一样。
主要有三个角色
[1]业务方：业务方主要在这里输入碎片化的目标或用户原声。然后开始调用AI
支持多种内容的输入，包括会议纪要，业务原声，文档等等
AI辅助方式
+ AI Agent提示词可以追问业务方内容，补全背景，约束与标准，最后得到优化后的业务方需求结果。

[2]产品同学：写 PRD、画原型、对齐进度
AI辅助方式
+ 可以看到业务方原始需求，并且AI Agent优化过后的需求结果
+ Agent 自动检索飞书文档、行业报告及竞品网站，对齐内外部信息，然后基于业务方提出的需求自动撰写PRD，画原型。
+ Agent 输出包含功能模块、业务流程、验收标准的落地产品方案，输出执行计划、资源预算及风险评估。

[3]技术同学：根据产品同学和业务方提出的需求内容做技术文档的撰写
AI辅助方式
+ 根据产品同学最后提交的内容，直接AI生成技术方案

[4]管理员端，可以看到整合所有信息，生成面向业务方的可视化HTML汇报页【AI生成】。

------

## 3. 角色职能变化

| **角色**     | **传统职能**             | **新范式职能**                                       |
| ------------ | ------------------------ | ---------------------------------------------------- |
| **产品同学** | 写 PRD、画原型、对齐进度 | **定义问题、发现价值、编排 AI 协作、审核方案质量**   |
| **AI Agent** | 简单的文本摘要、辅助写作 | **实时澄清、信息补全、知识整合、方案推演、原型生成** |
| **业务方**   | 提需求、写需求说明       | 提供背景与约束、确认业务价值是否闭环                 |
| **技术同学** | 评估排期、实现功能       | 负责可行性终审、系统边界定义、技术风险控制           |



## 4. 成功评估标准

- **交互深度**：Agent 是否能进行“高质量追问”，而非被动接受。
- **价值溢出**：Agent 是否发现了业务方未曾察觉的“新价值场景”。
- **落地程度**：输出的方案是否包含清晰的验收标准和资源预算，能否直接指导开发。
- **角色转变**：产品经理的工作时长是否更多分配给了“判断”和“设计流程”，而非“敲字”。
